[결정 사항] REQ-16 종결 — 3D 재구축으로 확정
------------------------------------------------------------------------
현 바닐라 JS + CSS 유사 3D 구현은 코어 루프 검증용 프로토타입으로
취급하고, 기획안 6장의 스택(TypeScript + React + Vite + R3F + Zustand)
으로 재구축한다. 근거:
  - 남은 작업 대부분(연출, 조명, UI)이 렌더러 종속 작업이라
    기능을 먼저 쌓을수록 재구축 비용이 커짐
  - 힐링 정원 장르에서 그래픽은 코어 루프의 일부 —
    '보는 맛' 검증이 안 된 상태
  - 게임 로직(성장 계산, 경제, 도감, 채집)은 렌더러 무관이라 이식 가능

원칙:
  - 기존 코드는 삭제하지 않는다. 먼저 현재 상태를 커밋(REQ-18)하고,
    새 구현은 별도 폴더(예: v2/) 또는 브랜치에서 시작한다.
  - 3D 최소 범위: 고정 쿼터뷰 카메라 + 무료 에셋(Kenney/Quaternius)
    + 섬 1개 + 밭 9칸. 카메라 조작·그림자 품질·후처리는 후순위.
  - 3D는 정원 씬 캔버스만 담당하고, 상점·인벤토리·도감 등 패널은
    일반 React 컴포넌트로 구현한다(R3F 표준 패턴).

탈출 조건: 2단계(REQ-21)에서 "에셋을 씬에 띄우고 클릭 처리"까지
1주 이상 막히면 재구축을 중단하고 바닐라 노선(v1 문서의 REQ-02, 03,
14 순서)으로 복귀한다. 이 경우에도 1단계 작업물은 그대로 유효하다.

1단계 — 재구축 전 수정 (현 코드베이스에서 수행, 이식 후에도 유효)
------------------------------------------------------------------------

REQ-18. 버전 관리 시작  [선행 필수]
  내용   : git 저장소에 커밋 이력이 없다. 현재 상태(문서 포함)를
           첫 커밋으로 남긴 뒤 아래 작업을 시작할 것. 이후 REQ 단위 커밋.

REQ-01. 오프라인 성장 상한이 "총 경과 시간"에 적용됨  [P0 버그]
  위치   : game.js getCropStatus() (약 917~928행)
  현상   : elapsed를 무조건 OFFLINE_GROWTH_CAP_MS(12시간)로 자르기 때문에
           성장 시간이 12시간을 넘는 작물은 계속 접속해 있어도 영원히
           완성되지 않는다. 티어4 작물(24시간+) 추가 시 게임이 깨진다.
  기대   : 상한은 "접속하지 않은 구간"에만 적용. state.lastSeenAt을
           활용해 접속 시점에 오프라인 초과분만큼 plantedAt을 밀어내는
           방식으로 변경하고, getCropStatus는 순수 경과 시간만 계산.
  완료기준: growMs 13시간 테스트 작물이 접속 유지 상태에서 정상 수확됨.

REQ-17. 시간 역행 방어 한 줄 추가  [P0에 준함, REQ-01과 같이 처리]
  내용   : 기기 시계가 뒤로 간 경우 plantedAt이 미래 값이 될 수 있다.
           로드 시 plantedAt = min(plantedAt, now) 보정 추가.
           (서버 시각 도입은 기획안 Phase 2에서 — 지금은 이 방어만)

REQ-13. 게임 데이터·로직을 화면 코드와 분리  [이식 준비의 핵심]
  내용   : 재구축의 다리 역할. 두 묶음으로 추출한다.
           (1) data: CROP_DEFS, FORAGE_DEFS, QUALITY_DEFS,
               PLOT_UNLOCK_COSTS, CODEX_REWARDS, 각종 시간 상수
           (2) logic: 상태 생성/로드/저장, getCropStatus, rollQuality,
               applyGatherRefill, applyDailyLogin, ensureDailyVisitor,
               경제(구매/판매/납품), 도감 집계
           이 모듈들은 DOM을 일절 참조하지 않아야 하며(문서 객체,
           innerHTML 금지), 기존 render* 함수에서만 import해 쓴다.
  완료기준: 분리 후 기존 게임이 동일하게 동작하고, logic 모듈을
           Node에서 단독 import해도 오류가 없다.

REQ-04. 손님 보상 배율 기획과 불일치 (2배 vs +50%)  [로직이라 지금 수정]
  위치   : game.js ensureDailyVisitor() bonus: 2, deliverVisitorOrder()
  기대   : 기획 수치(1.5배)로 통일하거나 기획안.md 밸런스 표를 2배로
           갱신해 문서·코드 일치. 납품 품질은 "낮은 품질 우선 소모"로
           변경하거나 선택 UI 제공 검토(UI 변경은 3단계로 미뤄도 됨).

REQ-05. 자정에 갱신된 손님이 저장되지 않음  [경미, 로직이라 지금 수정]
  내용   : ensureDailyVisitor()가 상태를 바꾼 경우 true를 반환하고
           호출부에서 saveState() 실행 (applyGatherRefill 패턴과 동일).

2단계 — R3F 재구축 (신규 항목)
------------------------------------------------------------------------

REQ-20. 프로젝트 세팅
  내용   : Vite + React + TypeScript 프로젝트 생성.
           의존성: three, @react-three/fiber, @react-three/drei, zustand.
           1단계에서 분리한 data/logic 모듈을 가져와 타입만 입힌다
           (동작 변경 금지). Zustand 스토어가 기존 state 객체를 감싸고
           localStorage 저장/로드는 기존 SAVE_KEY와 포맷을 그대로 사용.
  완료기준: 개발 서버가 뜨고, 기존 저장 데이터를 읽어 골드·씨앗 수치가
           화면에 표시된다.

REQ-21. 3D 정원 씬 최소 구현  [리스크 최대 구간 — 탈출 조건 적용]
  내용   : 순서대로, 각 소단계마다 확인하며 진행:
           (1) 고정 쿼터뷰 카메라 + 조명 + 바닥(섬) 배치
           (2) Kenney 또는 Quaternius 무료 팩에서 흙 타일·작물 모델
               로드(GLTF), 밭 9칸 그리드 배치
           (3) 밭 클릭 → 기존 handlePlotClick 로직 연결
           (4) 작물 성장 3단계를 모델 교체 또는 스케일로 표현,
               수확 가능 상태는 발광/부유 마커로 표시
           에셋은 한 팩으로 통일해 톤 유지(기획안 8장 리스크 대응).
  완료기준: 씨앗 선택 → 밭 클릭 심기 → 성장 → 수확이 3D 씬에서
           동작한다. 잠긴 밭·확장도 클릭으로 처리 가능.

REQ-22. UI 패널 React 이식
  내용   : 상점, 인벤토리, 도감, 오늘의 손님, 자원 스트립, 씨앗 바,
           토스트를 React 컴포넌트로 이식. 기존 HTML 구조·문안 재사용.
           v1의 REQ-02(매초 전체 재렌더)와 REQ-03(aria-live 스팸)은
           이 항목으로 대체된다 — React 상태 구독으로 바뀐 부분만
           렌더되므로 같은 문제를 만들지 않도록 다음을 지킬 것:
           - 남은 시간 표시는 1초 틱 상태를 구독하는 최소 컴포넌트로 한정
           - aria-live는 토스트에만 부여
  완료기준: 수확 펄스 등 애니메이션이 끊기지 않고, 버튼 포커스가
           시간 경과에도 유지된다.

REQ-23. 저장 호환 및 마이그레이션 골격  [v1 REQ-15 흡수]
  내용   : 기존 SAVE_KEY("mini-idle-garden-save-v1") 데이터를 그대로
           읽어 이어서 플레이 가능해야 한다. state.version 필드를
           실제로 사용하는 migrate(save) 함수 골격을 만들고, 이후
           스키마 변경(비료 등)은 version 증가 + 변환 함수로 처리.
  완료기준: v1에서 만든 저장으로 v2를 열었을 때 골드·밭·도감이 유지된다.

REQ-24. 구현 교체 및 정리
  내용   : v2가 REQ-21~23을 충족하면 루트의 index.html/game.js/styles.css
           를 v2 빌드로 교체(구 파일은 legacy/ 폴더로 이동, 삭제 금지).
           README를 새 실행 방법(npm install / npm run dev)으로 갱신.

3단계 — 새 기반 위 기능 구현 (v1의 P1 항목)
------------------------------------------------------------------------

REQ-06. "돌아온 동안 일어난 일" 웰컴백 요약
  내용   : 재접속 시 오프라인 경과 시간, 완성된 작물 수, 시든 작물 수,
           채집 리필 여부를 모달 카드 하나로 표시. 방치형의 핵심
           보상 체감 장치. (React 모달로 구현 — 3D 씬 위 오버레이)
  완료기준: 일정 시간 이상 떠났다 돌아오면 요약 카드가 1회 표시된다.

REQ-08. 물주기 기본 메커니즘
  내용   : 작물당 1회 무료 물주기(품질 확률 소폭 상승). 3D 씬에서
           물주기 시 물방울 연출. "접속 → 밭 돌보기" 루프의 핵심.
  완료기준: 심은 작물마다 물주기 1회 가능, rollQuality에 반영.

REQ-11. 수확 연출
  내용   : 수확 시 작물 팝(스케일) 애니메이션 + 획득 표시(+1 딸기).
           R3F 스프링 애니메이션 또는 간단한 스케일 트윈으로 충분.
  완료기준: 수확 클릭 시 3D 씬에서 시각 피드백이 재생된다.

4단계 — 콘텐츠 확장
------------------------------------------------------------------------

REQ-07. 채집물 → 비료 제작 루프
  내용   : 채집물의 사용처가 판매뿐인 문제 해결. 예: 버섯 2 + 들꽃 1
           → 비료 1, 사용 시 품질 확률 상승(물주기와 별개 슬롯).
           저장 스키마 변경이므로 REQ-23의 migrate로 version 처리.

REQ-09. 티어4 작물 및 장기 목표
  내용   : 달빛버섯/무지개꽃(24시간+, 희귀 씨앗). REQ-01 수정이
           전제 조건. 도감 보상(10칸)과 연결해 엔드 콘텐츠로.

REQ-10. 밤/낮 시각 연출
  내용   : 19시~5시에 씬 조명·하늘색 전환, 해→달 교체. R3F에서는
           조명 색·강도 보간으로 CSS보다 자연스럽게 구현 가능.
           밤 전용 채집물(밤빛 조각)의 존재를 시각적으로 안내.

REQ-12. 밸런스 프리셋 (데모 ↔ 실서비스)
  내용   : 데모(45초~10분)와 실서비스(4~12시간) 두 벌의 밸런스를
           data 모듈에 두고 상수 또는 쿼리 파라미터로 전환.
           "하루 1~2회 접속" 리듬은 실서비스 프리셋으로 지인 테스트.

REQ-19. 잔여 폴리시
  내용   : favicon, meta description, 모바일 성능 점검(로우폴리 유지,
           그림자 저해상도 — 기획안 8장). v1의 CSS 관련 항목
           (backdrop-filter 접두사, font-weight 정리)은 렌더러 교체로
           소멸 — 새 UI에서는 로드한 서체에 있는 웨이트만 사용할 것.
